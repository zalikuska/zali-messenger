# ZaliMessenger Design System

Этот документ описывает **только дизайн** ZaliMessenger: визуальный язык, UI-правила, компоненты, состояния и тон.
Его можно использовать как основу для новых проектов в том же стиле.

## 1. Суть стиля

Стиль Zali - это:

- dark-first интерфейс;
- lime как единственный яркий акцент;
- glassmorphism без перегруза;
- высокая плотность интерфейса;
- короткие, уверенные подписи;
- функциональность важнее декора.

Ключевой образ: **технический, собранный, не театральный**.

### Что должен чувствовать пользователь

- это современный инструмент, а не маркетинговая витрина;
- интерфейс умный, тихий и прямой;
- важные действия легко найти;
- ошибки и статусы видны сразу;
- форма и функция всегда на первом месте.

## 2. Визуальная ДНК

### Базовые принципы

1. Темный фон доминирует.
2. Lime используется точечно.
3. Поверхности выглядят как стекло с глубиной.
4. Рамки тонкие, свет мягкий.
5. Интерфейс должен быть плотным, но не тесным.
6. Никаких декоративных орбов, bokeh и лишних градиентных сцен.

### Запрещенные ощущения

- “лендинг” вместо рабочего интерфейса;
- декоративность без пользы;
- глянцевый SaaS-стиль с белыми панелями;
- перегруженные карточки с крупными декоративными блоками;
- слишком много разных акцентных цветов одновременно.

## 3. Цветовая система

### Основная палитра

| Роль | Цвет | Использование |
| --- | --- | --- |
| `Zali Lime` | `#cbff00` | primary actions, focus, active state, success accent |
| `Deep Dark` | `#090b0e` | основной фон |
| `Glass Base` | `rgba(255,255,255,0.03)` | фон карточек и панелей |
| `Glass Border` | `rgba(255,255,255,0.08)` | тонкие рамки |
| `Text Primary` | `#f2f2f2` / `#ffffff` | заголовки и основной текст |
| `Text Secondary` | `rgba(255,255,255,0.6)` | подсказки, метаданные |
| `Text Tertiary` | `rgba(255,255,255,0.25)` | disabled / subtle text |
| `Danger` | `#ff4d6d` | ошибки, destructive actions |

### Использование цвета

- Lime не должен появляться везде, только там, где нужен акцент.
- Красный - только для ошибок и разрушительных действий.
- Текст должен быть почти белым, не серым “на сером”.
- Фон не должен быть просто черным: нужна глубина и мягкая неоднородность.

### Фон

Фоновая среда обычно строится как:

- очень темная базовая заливка;
- мягкие подсветки lime;
- легкая сетка или шум;
- глубина через несколько слоев transparency.

## 4. Типографика

### Шрифтовой характер

- системный sans-serif;
- плотный, современный, без декоративности;
- заголовки - крупные и уверенные;
- body - спокойный и читабельный;
- монопространственный шрифт - для путей, кодов, хэшей, технических фрагментов.

### Типографические правила

- letter-spacing не должен быть отрицательным;
- заголовки могут быть жирными и капсом для брендинга;
- body не должен быть слишком мелким;
- в компактных панелях заголовки не должны быть hero-scale;
- строки текста должны вмещаться без налезания на соседние элементы.

### Рекомендуемые иерархии

| Роль | Характер |
| --- | --- |
| H1 | крупный, жирный, на главном экране |
| H2 / H3 | для секций и панелей |
| Body | обычный рабочий текст |
| Meta | подписи, пояснения, статусы |
| Mono | тех.данные, ID, коды, пути |

## 5. Форма поверхностей

### Карточки и панели

Карточка в стиле Zali:

- темная полупрозрачная поверхность;
- blur;
- тонкая рамка;
- мягкая тень;
- небольшой радиус;
- без ощущения “пластиковой” плитки.

### Рекомендованные характеристики

- background: `rgba(255,255,255,0.03)` или близко к этому;
- border: `1px solid rgba(255,255,255,0.08)`;
- radius: 12-18px;
- shadow: глубокая, мягкая, не грязная;
- backdrop blur: 16-40px, если платформа поддерживает.

### Графический язык

- панели выглядят как слои стекла;
- внутренние элементы не спорят с фоном;
- главное действие должно быть видно на расстоянии;
- вторичный контент должен оставаться тихим.

## 6. Сетка и компоновка

### Общая логика

Zali предпочитает:

- left sidebar + main content;
- верхнюю компактную шапку;
- плотно собранные блоки настроек;
- минимальное количество лишних экранов;
- понятное разделение между диалогами, чатом, настройками и статусом.

### Layout принципы

- рабочая область должна занимать максимум доступного пространства;
- sidebar должен быть узким, но полезным;
- панели в настройках - секционные, не “огромные плавающие карточки”;
- на широких экранах интерфейс должен чувствовать себя собранно;
- на узких экранах допускается переключение и скрытие вторичных панелей.

## 7. Кнопки

### Primary button

Использование:

- главный CTA;
- действие, которое пользователь должен заметить первым;
- подтверждение критичного рабочего шага.

Характер:

- фон lime;
- текст черный;
- легкое свечение;
- четкая визуальная приоритетность;
- без избыточной скругленности.

### Secondary button

Использование:

- вспомогательные действия;
- открыть настройки;
- обновить данные;
- менее важные переходы.

Характер:

- темная поверхность;
- тонкая рамка;
- спокойный hover;
- без кричащего цвета.

### Destructive button

Использование:

- удаление;
- revoke;
- сброс;
- закрытие с потерей данных.

Характер:

- красный акцент;
- простой и ясный текст;
- не смешивать с primary.

### Button rules

- кнопки должны быть читаемы с первого взгляда;
- hover должен ощущаться, но не прыгать;
- размеры кнопок должны быть стабильны;
- в кнопке лучше использовать иконку, чем длинный текст, если действие очевидно.

## 8. Поля ввода

### Inputs

Инпуты в Zali:

- темные;
- с аккуратной рамкой;
- без резкого контраста;
- с заметным focus состоянием;
- без лишних декоративных элементов.

### Правила

- placeholder должен быть коротким;
- поле не должно “плыть” по высоте;
- текст не должен упираться в границы;
- focus должен быть явно виден;
- ошибки должны показываться под полем, а не ломать layout.

### Формы

- формы компактные;
- поля выровнены по сетке;
- подсказки не должны конкурировать с вводом;
- state “disabled” должен выглядеть спокойным, а не сломанным.

## 9. Переключатели, тумблеры, чекбоксы

### Тогглы

Тумблеры используются там, где есть простая бинарная настройка.

Паттерн:

- off - нейтральный;
- on - lime-сигнал;
- label рядом должен объяснять, что именно включается;
- результат переключения должен быть очевиден сразу.

### Чекбоксы

Чекбоксы уместны в:

- настройках;
- списках опций;
- разрешениях;
- bulk selection.

Они должны быть маленькими, но легко заметными.

## 10. Списки, диалоги, навигация

### Sidebar list

Список диалогов / серверов:

- плотный;
- сканируемый;
- с понятным active state;
- с короткими подписями;
- с предсказуемой иерархией.

### Active item

Активный элемент должен:

- быть легко читаемым;
- иметь мягкий цветовой отклик;
- не выглядеть как отдельный яркий блок;
- сохранять общую стилистику.

### Empty states

Пустое состояние должно:

- объяснять, почему список пуст;
- говорить, что делать дальше;
- не выглядеть как ошибка;
- быть лаконичным.

## 11. Чат и сообщения

### Chat area

Чат должен ощущаться как:

- рабочий журнал;
- место для быстрой реакции;
- пространство с хорошей читаемостью.

### Message bubbles

Сообщения:

- не должны быть слишком “пузырчатыми”;
- должны иметь умеренное скругление;
- должны различаться по роли без чрезмерного контраста;
- между сообщениями должно быть мало пустоты, но достаточно воздуха.

### Media / attachments

Вложения должны:

- выглядеть как рабочие карточки, а не как декоративная галерея;
- иметь четкий статус загрузки;
- использовать lime только для завершенных / активных состояний.

### Reactions / status

Реакции и статусы:

- маленькие;
- не отвлекают от текста;
- должны быть видны при наведении или в состоянии активности;
- не превращают чат в “соцсеть”.

## 12. Заголовки, хедеры и статусные области

### Top bar

Верхняя полоса:

- компактная;
- функциональная;
- может показывать состояние сети / websocket / активного чата;
- не должна “съедать” высоту рабочей области.

### Status pills

Статусные pill-элементы:

- небольшие;
- слегка подсвеченные;
- используются для состояния соединения, режима и важных технических статусов.

## 13. Settings design

### Настройки как рабочая панель

Настройки Zali - это не длинная декоративная страница, а набор рабочих секций:

- профиль;
- стиль;
- сеть / аккаунт;
- logs;
- advanced.

### Секция

Каждая секция должна:

- иметь короткий заголовок;
- содержать ясное описание;
- не быть слишком высокой без причины;
- визуально отделяться, но не “выпирать”.

### Controls in settings

В настройках хорошо работают:

- radio / segmented buttons для тем;
- range sliders для числовых параметров;
- toggle switches для логических флагов;
- flat buttons для действий;
- compact help text под основным контролом.

## 14. Иконки и символы

### Принцип

Иконки должны быть:

- простыми;
- функциональными;
- не декоративными ради декоративности;
- хорошо считываться в малом размере.

### Где использовать

- attach;
- send;
- settings;
- search;
- close;
- status indicator;
- voice actions;
- file actions.

### Тон и форма

- предпочтительны знакомые символы и пиктограммы;
- если есть стандартная иконка, лучше взять ее;
- не усложнять контур без нужды.

## 15. Motion and feel

### Анимация

Движение должно быть:

- мягким;
- коротким;
- функциональным;
- без “показного” ease-интертеймента.

### Хорошие анимации

- hover lift;
- subtle border glow;
- panel fade in;
- active state transition;
- compact list highlight.

### Плохие анимации

- долгие и громкие переходы;
- bounce ради bounce;
- избыточные параллаксы;
- motion, который мешает чтению.

## 16. Tone of voice в интерфейсе

Текст в Zali должен быть:

- кратким;
- спокойным;
- уверенным;
- технически ясным;
- без лишнего пафоса.

### Хорошие примеры

- “Сообщение отправлено”
- “Контакт добавлен”
- “Подключение потеряно”
- “Не удалось сохранить”
- “Ключи подгрузятся из облака”

### Плохие примеры

- длинные маркетинговые фразы;
- чрезмерно дружелюбные формулировки;
- объяснения на полэкрана;
- многословные подсказки.

## 17. Дизайн для новых проектов

Если делать новый проект “в стиле Zali”, базовый рецепт такой:

1. Темный фон с мягкими lime-подсветками.
2. Glass-панели с blur и тонкой рамкой.
3. Основной CTA - lime button, остальные - subdued.
4. Рабочие списки и формы должны быть плотными и сканируемыми.
5. Настройки и инструменты - как аккуратные секции, а не как лендинг.
6. Ошибки и состояния всегда показывать явно.
7. Не смешивать Zali style с яркими градиентами, пастелью или “дружелюбным” SaaS-дизайном.

## 18. Quick checklist

Перед тем как считать интерфейс “в стиле Zali”, проверь:

- фон темный и не плоский;
- lime используется дозированно;
- карточки выглядят стеклянными;
- кнопки имеют ясную иерархию;
- текст читается на всех размерах;
- пустые состояния не молчат;
- ошибка видна сразу;
- layout не ломается от длинного текста;
- интерфейс ощущается как инструмент, а не витрина.

## 19. Сводка в одной фразе

**Zali - это dark glass, lime accents, compact utility, and zero visual fluff.**
