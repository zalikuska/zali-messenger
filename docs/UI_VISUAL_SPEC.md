# Zali Messenger — UI Visual Specification

## 1. Назначение
Единый визуальный язык интерфейса Zali Messenger: тёмная система с неоновым лимонным акцентом, стеклянными панелями и мягкими световыми ореолами. Стиль используется во всех основных экранах:
- авторизация;
- список контактов и поиск;
- окно чата;
- настройки;
- служебные панели и кнопки.

Цель — сделать интерфейс узнаваемым, контрастным и читаемым при низкой яркости фона.

---

## 2. Дизайн-токены

### 2.1 Цвета
Базовые CSS-переменные находятся в `Web/style.css`.

| Токен | Значение | Назначение |
|---|---:|---|
| `--bg` | `#090b0e` | Основной фон приложения |
| `--sidebar` | `rgba(11,13,16,.9)` | Фон левой панели |
| `--lime` | `#cbff00` | Базовый акцент, primary action, active state |
| `--lime-dim` | `rgba(203,255,0,.1)` | Фокусное кольцо и мягкая подсветка |
| `--lime-glow` | `rgba(203,255,0,.25)` | Яркая подсветка и светящиеся точки |
| `--lime-soft` | `rgba(203,255,0,.06)` | Общий фоновой ореол |
| `--text` | `#f2f2f2` | Основной текст |
| `--text2` | `rgba(255,255,255,.5)` | Вторичный текст |
| `--text3` | `rgba(255,255,255,.25)` | Подписи и малозаметные элементы |
| `--border` | `rgba(255,255,255,.07)` | Базовые границы |
| `--red` | `#ff4d6d` | Ошибки и destructive states |

### 2.2 Радиусы
| Токен/паттерн | Использование |
|---|---|
| `8px` | поля ввода, компактные кнопки, чипы |
| `12px` | основные поля, кнопки действий |
| `14px` | мини-карточки и превью вложений |
| `18px` | карточки настроек |
| `24px` | auth-card |
| `999px` | плашки, бейджи, круглые индикаторы |

### 2.3 Тени и свечение
- Стандартная карточка: `0 18px 48px rgba(0,0,0,.16)`.
- Акцентная подсветка: `0 0 0 3px var(--lime-dim)`.
- Primary glow: `0 0 18px rgba(203,255,0,.28)`.
- Световая точка online/active: `0 0 12px var(--lime-glow)`.

### 2.4 Поверхности
Поверхности строятся слоями:
1. **Базовый фон** — почти чёрный.
2. **Локальный цветной ореол** — мягкий lime radial gradient.
3. **Полупрозрачная плоскость** — `rgba(...,.02–.06)`.
4. **Тонкая рамка** — `var(--border)` или акцентный border при hover/focus.
5. **Inset highlight** — верхняя внутренняя подсветка `rgba(255,255,255,.04)`.

## 3. Принципы визуального языка

### 3.1 Контраст
- Акцентные действия всегда выполняются через лимонный цвет.
- Второстепенные элементы остаются в серой шкале.
- На тёмном фоне приоритет у текста и функциональных кнопок.

### 3.2 Слоистость
Каждый крупный блок имеет 2–3 уровня глубины:
- фон блока;
- граница;
- мягкая тень/ореол.

### 3.3 Hover/Focus
- Hover: лёгкий подъём `translateY(-1px)` и усиление рамки.
- Focus: `box-shadow: 0 0 0 3px var(--lime-dim)`.
- Active: краткий сдвиг вниз `translateY(1px)` только для интерактивных кнопок.

## 4. Компонентная модель

### 4.1 Titlebar
**Задача:** системная шапка окна и индикатор подключения.

Характеристики:
- высота: `40px`;
- фон: градиент с blur-эффектом;
- нижняя граница: `var(--border)`;
- индикатор соединения — `ws-pill`.

Состояния:
- `ws-pill.on` — зелёная точка и мягкое свечение;
- обычное состояние — красная точка.

### 4.2 Sidebar
**Задача:** навигация, контакты, поиск, информация о пользователе.

Структура:
- brand;
- search input;
- tools;
- suggestion dropdown;
- contacts list;
- own profile block (`.me`).

Поведение:
- список контактов скроллируется отдельно;
- активный контакт подсвечивается левой lime-полосой;
- hover использует светлый градиентный фон.

### 4.3 Contact card
**Состав:** аватар, имя, превью последнего сообщения, badge, remove button.

Состояния:
- default — прозрачный фон;
- hover — мягкая подложка и тень;
- active — inset lime bar слева + усиленная тень;
- badge — лимонный пилюльный счётчик.

### 4.4 Chat header
**Задача:** показать активный диалог и набор быстрых действий.

Характеристики:
- glass-like панель;
- нижняя граница;
- заголовок и подзаголовок в двух уровнях текста;
- служебные кнопки оформлены как нейтральные surface buttons.

### 4.5 Message bubble
**Входящие сообщения:**
- светлая полупрозрачная градиентная плашка;
- border `var(--border)`;
- тень для отделения от фона.

**Исходящие сообщения:**
- `var(--lime)`;
- тёмный текст `#050505`;
- border отключён как акцентный фон.

**Media / attachments:**
- `media-card` использует тёмную карточку для совместимости с картинками/видео;
- `file-chip` и `discord-media-shell` имеют свои радиусы и внутренние рамки.

### 4.6 Input area
Состоит из:
- блока вложений (`draft-attachments`);
- компоновки ввода (`input-bar`);
- кнопок attach/send.

Принципы:
- панель отделена верхней границей;
- input-bar — одна из ключевых surface-панелей;
- send-кнопка — primary action с лимонным фоном и glow.

### 4.7 Settings
`settings-card` — основная атомарная карточка в приложении.

Особенности:
- мягкий фон + легкий световой клин в углу (`::before`);
- карточки используются для логов, темы, масштаба, логики, опасных действий;
- тема выбирается через яркие `btn-theme`.

### 4.8 Auth modal
**Задача:** централизованный экран входа/регистрации.

Характеристики:
- fullscreen overlay с blur;
- карточка с радиальным lime-ореолом;
- один основной call-to-action и одна secondary action;
- поля анимируются последовательно при открытии.

## 5. Формы и состояния ввода

### Базовый input
- Высота: `38px` (контакты), `44px` (auth/settings).
- Border: `var(--border)`.
- Background: полупрозрачный тёмный.
- Focus: lime border + ring.

### Текстовые поля
- `auth-input`, `contact-input`, `settings-input` используют общий тёмный surface pattern.
- `settings-input` дополнительно может использовать моноширинный шрифт.

### Disabled
- opacity `0.55`;
- cursor `not-allowed`;
- отсутствие активной тени.

## 6. Анимации

| Анимация | Длительность | Назначение |
|---|---:|---|
| `suggest-drop` | `180ms` | раскрытие списка подсказок |
| `auth-pop` | `420ms` | появление auth-card |
| `auth-field-in` | `420ms` | последовательный вход полей |
| `shimmer` | `1.4s` | skeleton loading |

Правило: анимации должны оставаться короткими и не мешать работе; при `prefers-reduced-motion: reduce` они отключаются.

## 7. Responsive behavior

### Breakpoints
- `1100px` — перестройка настроек в одну колонку.
- `760px` — мобильная компоновка интерфейса.

### Правила
- Sidebar и topbar должны уплотняться без потери навигации.
- Settings переходят в один столбец.
- Отступы уменьшаются до компактных значений.
- Блоки сообщений и списка контактов сохраняют читаемость без горизонтального скролла.

## 8. Техническая реализация

### 8.1 Где живёт дизайн
Все токены и визуальные правила хранятся в:
- `Web/style.css`

### 8.2 Как расширять UI
При добавлении новых экранов и компонентов:
1. Использовать существующие CSS-переменные вместо новых “магических” цветов.
2. Соблюдать стек: фон → border → shadow → accent glow.
3. Не смешивать несколько акцентных цветов в одном action-сценарии.
4. Для интерактивных элементов использовать общие hover/focus паттерны.
5. Для больших панелей сначала задавать surface background, затем вводить внутренний glow через pseudo-element.

### 8.3 Что нельзя ломать
- lime как единый primary accent;
- читаемость текста на тёмном фоне;
- ограничение на количество ярких цветовых акцентов;
- отдельные scroll containers для длинных списков и логов.

---

## 9. Критерии качества
Дизайн считается согласованным, если:
- основные CTA и активные состояния выглядят одинаково во всех экранах;
- поля ввода имеют единый focus ring;
- карточки и панели используют общую поверхность и тень;
- акцент lime повторяется только в ключевых действиях и статусах;
- пользователь видит иерархию: header → list → content → action bar.
