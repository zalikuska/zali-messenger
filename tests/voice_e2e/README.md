# Voice call E2E test

Максимально правдивый end-to-end тест звонков: реальный `zali_server`, реальный
`Web/index.html`/`app.js`, два независимых Chromium-профиля с настоящим
fake-аудиоустройством (генерирует синтезированный тон, не тишину), настоящие
DOM-клики вместо вызова внутренних JS-методов.

## Установка (один раз)

```bash
cd tests/voice_e2e
npm install
npx playwright install chromium
```

## Запуск

```bash
cd tests/voice_e2e
node voice_call.test.js
```

Тест сам поднимает `cargo run -p zali_server` на свободном порту с изолированной
БД во временной директории, раздаёт `Web/` двум статическим серверам на разных
портах (разные origin = разные независимые сессии, как два разных устройства),
регистрирует двух пользователей, звонит, принимает, проверяет реальный RTP-трафик
в обе стороны, кладёт трубку, проверяет отклонение звонка — и убирает за собой
все процессы и временные файлы.

## Что именно проверяется

1. Принимающая сторона видит кнопки «Принять»/«Отклонить», а не сразу активный
   звонок (регрессия на баг с авто-подъёмом).
2. После клика «Принять» — SDP offer/answer доходит до `stable` на обеих сторонах.
3. Реальные аудио-пакеты (`RTCPeerConnection.getStats()`) идут в обе стороны —
   `packetsSent`/`packetsReceived` растут и совпадают между сторонами.
4. Независимая проверка через `AnalyserNode` — тот же код, что рисует шкалу
   «С сервера %» в UI — показывает ненулевой уровень декодированного звука на
   обеих сторонах. Это решающая проверка «звук реально дошёл и это не тишина».
5. Корректное завершение звонка (оба идут в `idle`).
6. Корректный сценарий отклонения звонка.

## Честные ограничения (см. комментарий в начале файла)

- Не проверяет TURN-релей и NAT — оба клиента на localhost, ICE соединяется
  напрямую.
- Fake-микрофон Chromium — не физическое железо.
- Бьёт по общему Web-коду (`interface.js`/`voice.rs`), который используют все
  клиенты, но не через нативный IPC-мост macOS/Windows-шеллов напрямую.
- `RTCPeerConnection.getStats().inbound-rtp.audioLevel` стабильно возвращает 0
  в headless Chromium несмотря на реальный трафик — это задокументированная
  особенность окружения (см. комментарий в тесте), решающая проверка звука
  идёт через независимый `AnalyserNode`-метр приложения, а не через эту метрику.
